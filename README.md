# Tutor Assistant

Текущая версия: **0.10.0**.

Десктоп-сервис для полного цикла подготовки материалов после занятия:

```text
запись микрофона и системного звука
→ локальная транскрибация faster-whisper
→ ручная проверка текста
→ публикация задания в students-26-27
→ генерация LaTeX, плаката и web-эквивалента через ChatGPT Work
```

Проект рассчитан на Windows, PowerShell и локальную обработку аудио. Аудиозаписи остаются на компьютере преподавателя. В репозиторий учеников передаются подтверждённый транскрипт и метаданные задания.

## Возможности MVP

- выбор ученика, предмета, темы и даты;
- выбор микрофона и входа системного звука/loopback;
- одновременная запись двух источников;
- сведение каналов через FFmpeg или локальный fallback;
- локальная транскрибация через `faster-whisper`;
- сырой текст, текст с таймкодами, сегменты JSON и очищенная версия;
- выделение реплик, указывающих на затруднения ученика;
- обязательное подтверждение транскрипта;
- SQLite-история и состояния заданий;
- создание `lesson.json` по формальной схеме;
- копирование задания в релевантную папку `students-26-27`;
- отдельная Git-ветка на каждое занятие;
- готовый prompt для запланированной задачи ChatGPT Work.

## Установка

```powershell
git clone https://github.com/ArtemLevin/tutor-assistant.git
cd tutor-assistant
uv sync --all-extras
uv run python scripts\bootstrap.py
```

`uv` создаёт `.venv`, фиксирует разрешённые версии в `uv.lock` и устанавливает GUI,
транскрибацию и инструменты разработки. Если доступен GNU Make, те же действия выполняются короче:

```powershell
make init
```

В Windows `make` можно запускать из Git Bash/MSYS2 или установить через привычный пакетный менеджер.
Все цели Make используют `uv`; активация виртуального окружения вручную не требуется.

Основные команды:

```powershell
make help
make run
make setup
make doctor
make check
make build
```

Прямые uv-эквиваленты работают и без Make:

```powershell
uv run --all-extras tutor-assistant-gui config\app.yaml
uv run --all-extras tutor-assistant --config config\app.yaml doctor
uv run --all-extras pytest -q
uv run --all-extras ruff check .
uv build
```

Проверьте FFmpeg:

```powershell
ffmpeg -version
ffprobe -version
```

## Конфигурация

Создайте рабочую конфигурацию:

```powershell
Copy-Item config\app.example.yaml config\app.yaml
```

Ключевой параметр:

```yaml
repository:
  students_repo: ../students-26-27
```

Путь может быть абсолютным:

```yaml
repository:
  students_repo: C:/Users/Артем/IdeaProjects/students-26-27
  auto_create_pr: true
  repository_full_name: ArtemLevin/students-26-27
  pr_base_branch: main
```

Рекомендуется хранить репозитории рядом, сохраняя их независимыми:

```text
C:\Users\Артем\IdeaProjects\
├── tutor-assistant\
├── students-26-27\
└── .tutor-assistant-worktrees\
```

`students-26-27` не следует помещать внутрь Git-репозитория `tutor-assistant`. Соседние каталоги
разделяют код приложения и материалы учеников, исключают вложенный `.git` и позволяют сервису
создавать изолированные worktree. Обе папки при необходимости можно открыть в одном workspace IDE.

Настройки аудиоконвейера:

```yaml
recording:
  sample_rate: 48000
  channels: 1
  mic_device: null
  system_device_id: null
  system_backend: soundcard
  chunk_seconds: 30
  diagnostics_seconds: 5
  queue_blocks: 256
  target_sample_rate: 48000
  dual_channel_transcription: true
  require_preflight: true
  silence_warning_seconds: 20
  device_timeout_seconds: 5
```

Настройки компилятора:

```yaml
latex:
  enabled: true
  auto_monitor: true
  engine: pdflatex
  latexmk_command: latexmk
  timeout_seconds: 180
  keep_build_files: false
  publish_pdf: true
  max_attempts: 2
  render_preview: true
  preview_dpi: 120
  poll_seconds: 60
```

Список учеников хранится в `config/students.yaml`. `id` должен совпадать с именем каталога внутри `students/`.

## Ученики и расписание

В расширенном режиме доступны вкладки «06 Ученики» и «07 Расписание».

При первом запуске приложение переносит минимальные сведения из `config/students.yaml` в таблицы
`crm_*` внутри `data/tutor-assistant.sqlite3`. YAML остаётся источником первоначальной миграции;
последующие изменения карточек сохраняются в локальной базе.

Карточка ученика содержит:

- ФИО, класс, школу, экзамен и образовательную цель;
- предметы, целевой балл, часовую зону и папку репозитория;
- ставку и локальные заметки;
- любое количество родителей или представителей;
- телефон, email, социальную сеть и предпочтительный канал связи.

Телефоны, email, социальные ссылки, ссылки на видеозвонки и заметки шифруются средствами Windows
DPAPI. Расшифровать их может тот же пользователь Windows на том же компьютере. Файл базы находится
в `data/`, которая исключена из Git.

Недельная сетка показывает понедельник–воскресенье и часы с 08:00 до 23:00. Двойной клик по пустой
ячейке создаёт занятие, по заполненной — открывает карточку. Доступны еженедельная серия, разовое
занятие, длительность 60/90/120 минут, ставка, тема и ссылка на подключение. Пересекающиеся серии и
разовые занятия блокируются.

Кнопка «Начать запись» в календарной карточке:

1. выбирает ученика и предмет;
2. переносит тему в быстрый запуск;
3. выполняет штатную аудиопроверку и отсчёт;
4. связывает запись с конкретным календарным занятием;
5. после сохранения выставляет календарный статус `completed`.

## Локальный домен контента ученика

Фундамент вкладки преподавателя «Контент ученика» хранит индекс занятий в той же локальной базе
`data/tutor-assistant.sqlite3`. SQLite содержит метаданные, статусы, ссылки на файлы и неизменяемые
версии транскрипта; сами WAV-файлы и `lesson.json` по-прежнему остаются в
`data/lessons/<lesson_id>/`.

Схема обновляется идемпотентными миграциями в таблице `schema_migrations`. При обновлении старой
базы существующие строки `lessons` сохраняются, а предмет и дата создания восстанавливаются из
JSON payload. Удаление занятия, файла или версии транскрипта является мягким: запись скрывается,
но может быть восстановлена.

Единственная прикладная точка CRUD — `StudentContentService`. Она предоставляет:

- фильтрацию по ученику, предмету, статусу, периоду и строке поиска;
- постраничную выдачу занятий;
- регистрацию и восстановление локальных файлов;
- историю версий транскрипта и возврат к выбранной версии;
- проверку, что любой изменяемый путь остаётся внутри рабочего каталога.

Перед первым открытием будущей GUI-вкладки можно безопасно проиндексировать старую структуру
каталогов. Повторный запуск не создаёт дубликаты:

```powershell
uv run tutor-assistant content-index
```

В расширенном режиме вкладка «08 Материалы» предоставляет read-only архив преподавателя:

- фильтры по ученику, предмету, статусу, периоду и строке поиска;
- постраничный список занятий, который загружается из SQLite в фоновом потоке;
- метаданные выбранного занятия и перечень известных локальных файлов;
- явные состояния «Доступен», «Файл отсутствует» и «Вне каталога данных»;
- чтение последней активной версии транскрипта из SQLite, даже если исходный TXT утрачен;
- встроенный аудиоплеер с play/pause, перемоткой и скоростью от 0,75× до 2×;
- выбор смешанной дорожки, голоса преподавателя или ученика, если файлы доступны;
- переход к распознанным сегментам по временным меткам;
- открытие остальных доступных файлов штатным приложением Windows;
- переход к материалам конкретного ученика из его CRM-карточки.

Обычная кнопка «Обновить» не сканирует файловую систему. Полная проверка
`data/lessons/<lesson_id>` выполняется в фоне при первом открытии вкладки или по отдельной кнопке
«Синхронизировать каталог».

Воспроизведение использует единую аудиосессию приложения. Перед началом новой записи плеер
автоматически останавливается, а во время активной записи и сохранения старое аудио заблокировано,
чтобы оно не попало в WASAPI Loopback. Закрытие карточки занятия или уход со вкладки также
останавливает плеер. Отсутствующий, повреждённый или неподдерживаемый аудиофайл отображается как
ошибка в интерфейсе и не завершает работу приложения.

Кнопка «Создать / импортировать» добавляет в архив занятие без запуска записи. В форме можно:

- создать пустую карточку занятия вручную;
- импортировать WAV, MP3, M4A, AAC, FLAC или OGG;
- импортировать готовый UTF-8 транскрипт в формате TXT или Markdown;
- сразу поставить импортированное аудио в штатную очередь транскрибации.

Импорт выполняется в фоновом потоке. Файлы сначала копируются во временный staging-каталог,
проверяются и только затем атомарно переносятся в `data/lessons/<lesson_id>`. Поэтому исходный файл
можно удалить после успешного импорта. При отмене или ошибке временные файлы и незавершённая запись
SQLite удаляются. Повторный импорт того же аудио блокируется по SHA-256, даже если файл был
переименован.

## Работа с приложением

### 1. Подготовка занятия

Выберите:

- ученика;
- предмет;
- тему;
- дату;
- микрофон;
- устройство воспроизведения, через которое слышно ученика.

Для вашей конфигурации рекомендуются:

- `Microphone (fifine Chat) [Windows WASAPI]` — ваш голос;
- `Динамики (G733 Gaming Headset) [WASAPI LOOPBACK]` — голос ученика и другой звук из G733.

Перед тестом запустите воспроизведение звука или попросите ученика говорить. При тишине индикатор
системного звука закономерно останется на нуле. Выбор автоматически сохраняется в конфигурации.

### 2. Запись

Нажмите «Начать запись». Приложение создаст:

```text
data/lessons/<lesson_id>/recording/
├── session.json
├── chunks/
│   ├── microphone/
│   │   ├── mic_00000.wav
│   │   └── mic_00001.wav
│   └── system/
│       ├── system_00000.wav
│       └── system_00001.wav
├── microphone.wav
├── system.wav
├── lesson.wav
├── sync_report.json
└── audio_quality_report.json
```

Микрофон и системный звук сохраняются раздельно. После завершения создаётся сведённый `lesson.wav`.

Audio callback выполняет только копирование блока в очередь. Дисковые операции работают в двух
writer-потоках. GUI показывает заполнение очередей, потерянные блоки и максимальную задержку
записи. `sync_report.json` содержит частоты устройств, смещение старта, длительности, оценку
дрейфа и применённые коэффициенты коррекции.

`audio_quality_report.json` содержит показатели обеих дорожек и поле `ready`. Перед началом занятия
кнопка проверки создаёт короткую запись в `data/diagnostics/`, после чего доступны отдельные кнопки
прослушивания микрофона и звука ученика.

После аварийного завершения соберите пригодные чанки:

```powershell
tutor-assistant-recover data\lessons\<id>\recording
```

Также можно выбрать уже готовый аудиофайл и перейти сразу к транскрибации.

### 3. Транскрибация

Кнопка «Запустить локальную транскрибацию» загружает модель `small` и создаёт:

```text
transcript/
├── 00_raw_whisper.txt
├── 00_raw_timestamped.txt
├── 00_raw_segments.json
├── 03_content_only_medium.txt
├── transcript_verified.txt
├── important_student_signals.json
├── teacher_transcript.txt
├── teacher_segments.json
├── student_transcript.txt
├── student_segments.json
└── manifest.json
```

Первый запуск модели может занять дополнительное время.

Для записи, созданной приложением, дорожки транскрибируются раздельно:

```text
[Преподаватель] Почему эта точка выколотая?
[Ученик] Потому что она обращает знаменатель в ноль.
```

Сигналы затруднений в таком режиме извлекаются из реплик ученика.

### 4. Проверка

Откройте вкладку «Транскрипт», исправьте числа, имена и формулы, затем нажмите «Подтвердить транскрипт».

### 5. Публикация

Приложение:

1. обновляет сведения о ветке `main` репозитория учеников;
2. создаёт изолированный временный worktree;
3. создаёт ветку вида `lesson/<student>-<date>-<id>`;
4. формирует каталог занятия;
5. создаёт commit и отправляет ветку в GitHub;
6. удаляет временный worktree после публикации.

При доступном GitHub CLI приложение создаёт draft PR. Проверка и авторизация:

```powershell
gh auth status
gh auth login
```

URL сохраняется в `lesson.json`; открыть PR можно на вкладке публикации.

В `students-26-27` появится:

```text
students/<student>/lessons/<date_topic>/
├── lesson.json
├── job.status.json
└── source/
    ├── transcript.txt
    ├── transcript_generated.txt
    ├── transcript_timestamped.txt
    ├── segments.json
    ├── important_student_signals.json
    └── transcription_manifest.json
```

## ChatGPT Work

Готовый текст задачи расположен в:

```text
automation/chatgpt-work-prompt.md
```

Рекомендуемая настройка:

- тип: самостоятельная Scheduled task;
- период проверки: один раз в час;
- доступ: GitHub plugin с правом записи и image generation;
- репозиторий: `ArtemLevin/students-26-27`;
- источник правил: полный prompt из `automation/chatgpt-work-prompt.md`;
- результат: commit в существующую ветку и draft PR занятия.

Scheduled task на web работает через подключённый GitHub и не сохраняет локальную папку между
запусками. Каждый законченный этап должен быть сразу отправлен в GitHub. Для локального режима
Desktop-приложение ChatGPT и компьютер должны оставаться включёнными; рекомендуется выделенный
worktree.

Перед включением расписания выполните prompt вручную для одного тестового draft PR и проверьте
права GitHub, создание PNG и запись файлов в head-ветку. Первые запуски расписания также следует
проверять вручную.

### Как агент находит занятие

Новые задания находятся в head-ветках открытых draft PR, имя которых начинается с `lesson/`.
До слияния PR этих файлов нет в `main`. Поэтому агент:

1. получает открытые draft PR;
2. отбирает ветки `lesson/*`;
3. читает `job.status.json` из head-ветки PR;
4. обрабатывает одно самое старое задание;
5. повторно проверяет head SHA перед записью;
6. создаёт commit в той же ветке.

`job.status.json` является источником состояния между Tutor Assistant и ChatGPT Work.
`lesson.json.status` используется как описание занятия и может отставать на один этап.

### Этапы агентной обработки

| Статус | Исполнитель | Действие |
|---|---|---|
| `ready_for_generation` | ChatGPT Work | Создать и проверить только TEX, установить `generated_tex` |
| `generated_tex` | Tutor Assistant | Обнаружить новую версию TEX и начать локальную компиляцию |
| `compiling_pdf` | Tutor Assistant | Выполнить `latexmk`, отчёт и PNG-предпросмотр |
| `compile_failed` | ChatGPT Work | Исправить блокирующие ошибки TEX и вернуть `generated_tex` |
| `pdf_review_required` | ChatGPT Work | Создать плакат, web-страницу, обновить `index.html` и проверки |
| `completed` | Преподаватель | Просмотреть draft PR и выполнить merge |

Один запуск агента выполняет один этап одного занятия. Такое ограничение предотвращает смешивание
контекста учеников и позволяет безопасно повторять прерванную задачу.

После создания TEX Tutor Assistant публикует PDF в ту же ветку. При ошибке компиляции он добавляет
`reports/latex/latex_fix_request.md` и `reports/latex/compilation.log`. Следующий запуск ChatGPT Work
исправляет TEX в рамках доступного количества попыток.

### Мгновенный запуск

Tutor Assistant не вызывает конкретный чат напрямую. Scheduled task обнаруживает новые задания на
очередной проверке. Для запуска сразу после публикации потребуется отдельный API-worker, вызываемый
Tutor Assistant или GitHub Actions. Такой worker должен самостоятельно реализовать доступ к GitHub,
генерацию изображений, контроль состояний и хранение API-ключа в secrets.

## Локальная компиляция PDF

Проверить окружение:

```powershell
tutor-assistant latex-doctor
```

Ожидаемые команды:

```powershell
latexmk --version
pdflatex --version
kpsewhich tikz.sty
kpsewhich tkz-euclide.sty
pdftoppm -h
```

Скомпилировать отдельный файл:

```powershell
tutor-assistant compile C:\lessons\12.07.26.tex
```

Результаты сохраняются рядом с пособием:

```text
handbook/
├── 12.07.26.tex
├── 12.07.26.pdf
├── build/
│   ├── compilation.log
│   ├── compilation.json
│   └── latex_fix_request.md
└── preview/
    ├── page-1.png
    └── page-2.png
```

Скомпилировать TEX непосредственно в опубликованной ветке занятия:

```powershell
tutor-assistant compile-remote data\lessons\<id>\lesson.json
```

Найти занятия, где появился новый TEX:

```powershell
tutor-assistant scan-latex
```

В GUI откройте вкладку «4. PDF». Там доступны:

- проверка TeX Live;
- ручной выбор `.tex`;
- просмотр журнала;
- открытие PNG-страниц;
- ручная проверка удалённых веток;
- фоновый мониторинг с интервалом из `latex.poll_seconds`.

### Безопасность компиляции

Компилятор блокирует:

```latex
\write18
\immediate\write18
\openin
\openout
\input{|command}
\input{/absolute/path}
\input{../outside}
```

Компиляция всегда выполняется с `-no-shell-escape`, `openin_any=p`, `openout_any=p`, во временном
каталоге и с заданным тайм-аутом. Исходный каталог занятия передаёт только ресурсы, расположенные
рядом с пособием.

## CLI

Полная диагностика в удобном для чтения виде:

```powershell
make doctor
```

Для CI или автоматизации:

```powershell
make doctor-json
make doctor-strict
make support
```

`doctor-strict` возвращает код `1`, когда обязательный компонент отсутствует. Обычный `doctor`
показывает тот же отчёт и позволяет продолжить настройку.

Журналы приложения находятся в `data/logs/`. Диагностический ZIP можно создать командой:

```powershell
uv run tutor-assistant --config config\app.yaml support-bundle
```

Архив содержит сведения об окружении, устройствах, безопасные настройки, отчёты последних записей
и журналы. Аудиофайлы и транскрипты в архив не включаются.

Показать микрофоны и WASAPI Loopback-выходы:

```powershell
uv run tutor-assistant devices
```

Создать занятие:

```powershell
tutor-assistant create `
  --student nikol_sarkisyants `
  --subject mathematics `
  --topic "Логарифмические неравенства"
```

Транскрибировать готовое аудио:

```powershell
tutor-assistant transcribe data\lessons\<id>\lesson.json C:\lessons\lesson.wav
```

Опубликовать подтверждённое занятие:

```powershell
tutor-assistant publish data\lessons\<id>\lesson.json
```

## Состояния задания

```text
draft
→ recording
→ recorded
→ transcribing
→ review_required
→ ready_for_generation
→ published
→ generated_tex
→ compiling_pdf
→ pdf_review_required
→ generating
→ completed
```

Ошибка LaTeX переводит занятие в `compile_failed`. После исправления новой версии TEX начинается
следующая попытка с `generated_tex`. Для остальных исключений используется `failed`, а причина
сохраняется в `lesson.json`.

## Проверка проекта

```powershell
make check
```

Цель проверяет `uv.lock`, Ruff, форматирование, тесты и компиляцию Python-модулей. Отдельные этапы:

```powershell
make lock-check
make lint
make format-check
make test
```

Проверить конкретный `lesson.json`:

```powershell
python scripts\check_job.py path\to\lesson.json
```

Если рядом находится `job.status.json`, команда автоматически проверит его по
`schemas/job-status.schema.json`. Старые маркеры только с полем `status` продолжают поддерживаться.

## Ограничения версии 0.7.0

- устройство воспроизведения должно быть подключено до запуска приложения;
- после подключения или отключения гарнитуры перезапустите Tutor Assistant для обновления списка;
- коррекция дрейфа автоматически применяется для расхождений до двух секунд; более крупные
  расхождения фиксируются в отчёте и требуют проверки устройств;
- спорные формулы проверяются преподавателем в сегментном редакторе;
- автоматический draft PR требует установленный и авторизованный GitHub CLI;
- визуальная проверка сложной вёрстки остаётся за преподавателем; приложение создаёт PNG-предпросмотр;
- Scheduled task обнаруживает задания при периодической проверке; немедленный внешний запуск
  ChatGPT Work через HTTP отсутствует.

## Безопасность данных

- `data/`, аудио и локальная SQLite-база исключены из Git;
- в ученический репозиторий отправляется подтверждённый текст;
- перед публикацией следует удалить случайно распознанные персональные или технические фрагменты;
- токены GitHub и другие секреты не следует хранить в YAML-файлах проекта.
